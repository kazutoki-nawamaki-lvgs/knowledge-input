# ボタンコンポーネントはなぜ難しいのか

No. 002
カテゴリ: コンポーネント

## 今日の問い

小さな部品に見えるボタンに多くの設計判断が集まるのはなぜか

## 先に結論

**ボタンコンポーネントが難しいのは、見た目の部品でありながら、操作の優先度、状態、リスク、アクセシビリティ、実装責務が一箇所に集まるから**です。

ボタンは「押せる四角」を作れば終わる部品ではありません。ユーザーに次の行動を示し、処理中であることを伝え、危険な操作を区別し、キーボードやスクリーンリーダーでも使えるようにし、フォーム送信や画面遷移との責務を混ぜすぎないようにする必要があります。

つまりボタンは、デザインシステムの中で最も小さく見えながら、チームのUI判断が最も露出しやすい部品です。**良いボタン設計とは、色やサイズを揃えることではなく、操作にまつわる判断を再利用できる形にすること**です。

## 開発者としての入口

SaaSプロダクトでは、ボタンはほとんどすべての画面に出てきます。保存、作成、検索、招待、削除、キャンセル、再送信、エクスポート、承認、却下。どれもボタンで表現できそうですが、それぞれユーザーに求める行動の重さは違います。

開発者にとって厄介なのは、ボタンの差分が見た目だけでは終わらないことです。`primary` や `secondary` のようなバリアント、`disabled` や `loading` のような状態、`type="submit"` のようなHTMLの振る舞い、アイコンだけのボタンのアクセシブルな名前、破壊的操作の扱い。これらを画面ごとに個別実装すると、見た目は似ていても挙動や使い方が少しずつずれます。

デザインシステムにおけるボタンは、単なる共通UIではありません。**プロダクト内の操作を、同じ強さ、同じ状態、同じルールで扱うための入口**です。だからこそ、最初に雑に作ると後から修正範囲が広くなり、逆に最初から過剰に抽象化すると使いにくいコンポーネントになります。

## 本文

### ボタンは、ユーザーに「何をすればよいか」を示す

ボタンの第一の役割は、ユーザーに実行可能な行動を示すことです。保存ボタンが目立つのは、単に色を付けたいからではありません。その画面でユーザーが次に進むための主要な行動を、視覚的に見つけやすくするためです。

ここで重要なのは、すべてのボタンを同じ強さにしないことです。たとえば設定画面に「保存」「キャンセル」「削除」が並ぶ場合、この3つは同じ「押せる操作」でも意味が違います。保存は主操作、キャンセルは補助操作、削除は危険な操作です。これらを同じ色、同じ強さ、同じ位置関係で並べると、ユーザーはどれが推奨される行動なのかを毎回判断しなければなりません。

だからボタン設計では、`primary`、`secondary`、`tertiary`、`destructive` のようなバリアントが必要になります。ただし、バリアントは色の名前ではありません。**バリアントとは、操作の優先度やリスクをUI上の強弱に変換するための判断基準**です。

よくある失敗は、見た目の都合でバリアントを増やすことです。「青が強すぎるから薄い青」「枠線が欲しいから別バリアント」「この画面だけ少し小さくしたい」と増やしていくと、後から誰も使い分けを説明できなくなります。バリアントを追加するときは、「この操作は既存の判断基準では表現できないのか」を先に確認する必要があります。

ここで押さえることは、**ボタンの見た目は、操作の意味から逆算して決める**という点です。色や枠線の違いだけで設計すると、デザインシステムは使い方を支えられません。

### 状態は、押せるかどうか以上の情報を伝える

ボタンには多くの状態があります。通常状態、hover、active、focus、disabled、loading、selected、pressed。見た目だけで考えると細かい差分に見えますが、実務ではそれぞれが違う意味を持ちます。

特に混同しやすいのが `disabled` と `loading` です。`disabled` は、今その操作を実行できないことを示します。一方で `loading` は、操作はすでに受け付けられ、処理中であることを示します。どちらも「押せない」状態に見えますが、ユーザーに伝えるべき情報は違います。

たとえばフォームの保存ボタンを考えます。必須項目が未入力なら、保存ボタンを無効にするか、押したあとにエラーを出すかを決める必要があります。保存ボタンを押したあと通信中なら、二重送信を防ぎつつ、処理が進んでいることを示す必要があります。通信に失敗したら、ボタンを戻すだけでなく、ユーザーが次に何をすればよいかを画面上で伝えなければなりません。

ここを画面ごとに任せると、ある画面では `disabled` の理由が分からず、別の画面ではローディング中も連打でき、さらに別の画面では失敗しても何も表示されない、といった差が出ます。ユーザーから見ると、同じプロダクト内で操作の信頼性が揺れているように感じられます。

**状態設計の目的は、ボタンの表情を増やすことではなく、操作が今どういう段階にあるのかを一貫して伝えること**です。状態の種類を決めるときは、ユーザーに伝える意味と、実装が持つ責務をセットで考える必要があります。

### アクセシビリティは、後付けではなくボタンの成立条件である

ボタンはHTMLに `<button>` があるため、簡単にアクセシブルにできるように見えます。実際、正しく使えば `<button>` はキーボード操作、フォーカス、スクリーンリーダーとの相性がよい要素です。しかし、実務ではここから外れる実装が多くあります。

たとえば、`div` にクリックハンドラを付けてボタンのように見せる、フォーカスリングを消す、アイコンだけのボタンにラベルを付けない、`disabled` にした理由を画面上で説明しない、ローディング中の変化が視覚以外では伝わらない。どれも見た目のレビューだけでは見落とされやすい問題です。

アイコンだけの削除ボタンを例にすると、見える人にはゴミ箱アイコンで意味が伝わるかもしれません。しかしスクリーンリーダーには、アクセシブルな名前がなければ「ボタン」としか伝わらない可能性があります。さらに、同じ一覧に削除ボタンが複数あるなら、「削除」だけではどの行を削除するのか分かりません。見た目では省略できる情報でも、操作として成立させるには名前や文脈が必要になります。

このため、デザインシステムのボタンにはアクセシビリティ要件を含めるべきです。アイコンのみの場合はラベルを必須にする。フォーカス表示は消さない。`button` と `a` の使い分けを決める。ローディングやエラーはボタン単体ではなく周辺UIと合わせて伝える。こうした条件がないと、各画面の実装者が毎回アクセシビリティ判断を背負うことになります。

**アクセシビリティは、ボタンに後から足す品質ではなく、ボタンが操作部品として成立するための最低条件**です。デザインシステムに含めることで、善意ではなく標準の振る舞いとして扱えるようになります。

### 責務を広げすぎると、便利な部品が壊れやすくなる

ボタンコンポーネントを作るとき、便利にしようとして責務を詰め込みすぎることがあります。権限判定、確認ダイアログ、API呼び出し、トースト通知、画面遷移、ローディング管理まで `Button` に持たせると、最初は使う側が楽に見えます。

しかし、ボタンの責務が広がりすぎると、再利用しにくくなります。ある画面では確認ダイアログが必要だが別の画面では不要、ある操作では成功後に一覧へ戻るが別の操作では同じ画面に残る、ある権限では非表示にしたいが別の権限では disabled にして理由を出したい。こうした差分をすべて `Button` の props で吸収しようとすると、コンポーネントは複雑な条件分岐の塊になります。

デザインシステム上の `Button` が主に持つべき責務は、見た目、基本的な状態、アクセシブルな操作、サイズやバリアントの一貫性です。業務ロジックや通信処理は、原則として画面側、フォーム側、または別の振る舞いコンポーネントに置くほうが扱いやすくなります。

もちろん、すべてを完全に分ければよいわけではありません。たとえば `loading` 表示や二重送信を防ぐための `disabled` 連動は、ボタンの使い勝手に深く関わります。大切なのは、**Button が操作の表現を担当するのか、操作の実行そのものを担当するのかを混ぜないこと**です。

ここでの判断基準はシンプルです。その責務がプロダクト全体のボタンに共通するなら `Button` に寄せる価値があります。一方で、特定画面の業務ルールや成功後の振る舞いなら、ボタンの外に置くほうが変更に強くなります。

## 実務への接続

### まず、バリアントを意味で説明できるかを見る

自分たちのSaaSで最初に確認したいのは、ボタンのバリアントをチームが意味で説明できるかどうかです。`primary` は何に使うのか、1画面に複数あってよいのか、`destructive` は色だけでなく確認や配置のルールを伴うのか、リンク風ボタンはいつ使うのか。ここが曖昧だと、レビューは好みの議論になりやすくなります。

ボタン一覧を作るだけでは不十分です。各バリアントに「使う場面」「避ける場面」「代表例」を短く書くと、デザイナーとエンジニアが同じ言葉で判断できます。特に destructive と disabled は、見た目だけでなくユーザーの誤操作や迷いに直結するため、早めに基準を持つ価値があります。

### 状態の抜けは、フォームと非同期処理で見つかりやすい

状態設計の抜けは、静的なコンポーネント一覧だけでは見つかりにくいです。実際の画面、特にフォーム送信や非同期処理を含む画面で確認すると、問題が見えやすくなります。

保存を押した直後に何が起きるか。連打できるか。処理中にラベルは変わるか。失敗したときボタンは戻るか。`disabled` の理由はユーザーに分かるか。キーボードだけでフォーカス位置を追えるか。こうした確認をすると、ボタン単体の設計と周辺UIの責務分担が見えてきます。

デザインシステムのドキュメントには、静的な見た目だけでなく「送信中」「失敗後」「権限不足」「入力不足」のような実務上の状態を含めると、実装時の判断が安定します。

### レビュー観点をコンポーネントに埋め込む

ボタンの品質を毎回レビュー担当者の注意力に頼ると、抜け漏れが起きます。だからこそ、できるものはコンポーネントや型に寄せ、できないものはドキュメントとレビュー観点に残します。

たとえばアイコンだけのボタンでは `aria-label` 相当のラベルを必須にする、`variant` と `size` を許可された組み合わせに絞る、`as="a"` と `button` の使い分けを明示する、`disabled` に頼りすぎないフォーム設計をガイドに書く。こうすると、レビューで毎回ゼロから指摘しなくても、標準の使い方に乗りやすくなります。

重要なのは、厳しく縛ること自体ではありません。**よく起きるミスを、個人の注意ではなく仕組みで減らすこと**です。ボタンは利用頻度が高い分、ここを整える効果が大きく出ます。

## 今日のまとめ

### 小さい部品ほど、判断が凝縮される

ボタンは小さなUI部品ですが、そこには操作の優先度、状態、リスク、アクセシビリティ、責務分離が集まります。見た目だけを共通化しても、使い方の判断が揃わなければ、プロダクト全体の体験は安定しません。

**ボタン設計で再利用したいのは、ボタンの形ではなく、操作をどう扱うかという判断**です。

### バリアントと状態は、意味から設計する

`primary` や `destructive` は色の違いではなく、操作の意味を表す言葉です。`disabled` や `loading` も、押せない見た目の違いではなく、ユーザーに伝えるべき状態の違いです。

ボタンを設計するときは、「どう見えるか」だけでなく、「ユーザーはこの操作をどう理解するか」「この状態で次に何をすればよいか」まで含めて考える必要があります。

### 責務の境界を決めると、長く使える

便利な `Button` を作ろうとして業務ロジックまで詰め込むと、画面ごとの差分に耐えにくくなります。共通コンポーネントとして持つべき責務と、画面側に残すべき責務を分けることが、変更に強いデザインシステムにつながります。

ボタンは最初に作る部品であることが多いからこそ、ここで決めた考え方が他のコンポーネントにも波及します。

## 自分のプロダクトに引きつける問い

- `primary`、`secondary`、`destructive` などのボタンバリアントを、色ではなく操作の意味で説明できるか？
- フォーム送信や非同期処理で、`disabled`、`loading`、エラー後の戻り方が画面ごとにずれていないか？
- アイコンだけのボタン、危険な操作のボタン、権限不足時のボタンは、キーボードやスクリーンリーダーでも意味が伝わるか？
- `Button` コンポーネントに、特定画面の業務ロジックや通信後の振る舞いまで持たせすぎていないか？
