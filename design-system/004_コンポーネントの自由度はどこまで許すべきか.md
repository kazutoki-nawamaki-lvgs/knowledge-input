# コンポーネントの自由度はどこまで許すべきか

No. 004
カテゴリ: 設計

## 今日の問い

再利用性と柔軟性のバランスはどこで崩れるのか

## 先に結論

コンポーネントの自由度は、高ければ高いほどよいわけではありません。**良いコンポーネントAPIは、使う人に自由を渡す前に、プロダクトとして守りたい判断を先に閉じ込めます。**

再利用性を高めようとして何でも変更できるようにすると、最初は便利に見えます。しかし、色、余白、サイズ、状態、配置、振る舞いを個別画面が自由に上書きできるようになると、同じコンポーネントを使っていても体験は揃わなくなります。

大事なのは、自由をなくすことではありません。**変えてよい部分と、変えてはいけない部分を設計として分けること**です。コンポーネントの柔軟性は、利用側の都合に無制限に合わせることではなく、プロダクトの一貫性を壊さずに必要な差分を表現できることです。

## 開発者としての入口

SaaSプロダクトでは、同じUI部品が多くの画面で使われます。ボタン、入力欄、カード、テーブル、モーダル、メニュー、アラート。これらを共通コンポーネントにすると、開発速度は上がり、見た目も揃いやすくなります。

一方で、実際の画面には例外が出ます。「この画面だけボタンを少し小さくしたい」「このカードだけ余白を詰めたい」「このテーブルだけ行クリックではなくチェックボックスを使いたい」。そのたびに props を増やすと、コンポーネントはすぐに大きくなります。

開発者にとって難しいのは、要望を断つことではなく、どの要望をコンポーネントのAPIとして受け止め、どの要望を別の表現や別コンポーネントに分けるかを判断することです。ここを曖昧にすると、共通化したはずの部品が、画面ごとの都合を集めた巨大な分岐になります。

## 本文

### 自由度が高いほど、判断が利用側へ漏れる

コンポーネントに多くの props を用意すると、利用側は細かく調整できます。`size`、`variant`、`tone`、`align`、`dense`、`fullWidth`、`showIcon`、`customClassName` のような選択肢が増えれば、さまざまな画面に対応しやすく見えます。

しかし、自由度が増えるほど、利用側は毎回判断しなければなりません。この操作は primary なのか secondary なのか。このフォームは compact でよいのか。この余白は画面都合で詰めてもよいのか。つまり、コンポーネントが吸収すべき判断が、画面実装へ戻ってしまいます。

**再利用性とは、同じ見た目を何度も呼び出せることではなく、同じ判断を何度も考え直さなくてよいこと**です。props が多すぎるコンポーネントは、一見再利用されていても、実際には判断の再利用に失敗していることがあります。

ここで押さえることは、自由度そのものが悪いのではなく、自由度の置き場所が重要だという点です。プロダクトとして揃えるべき判断まで利用側に渡すと、デザインシステムは部品集にはなっても、品質を支える仕組みにはなりません。

### 代表例として、Button の自由度を見る

たとえば `Button` は、自由度の設計が崩れやすいコンポーネントです。最初は `variant="primary"` と `variant="secondary"` だけで十分に見えます。ところが実装が進むと、「危険な操作用の赤いボタンがほしい」「一覧内では小さくしたい」「アイコンだけにしたい」「横幅を親に合わせたい」「テキストリンク風にしたい」といった要望が出てきます。

これらをすべて1つの `Button` に入れると、使う側は便利になります。しかし、`variant` と `size` と `tone` と `iconOnly` と `fullWidth` と `asLink` の組み合わせが増えるほど、何が許された使い方なのか分かりにくくなります。赤い primary ボタン、アイコンだけの destructive ボタン、リンク風の保存ボタンのように、プロダクトとして意味が曖昧な組み合わせも生まれます。

このとき必要なのは、単に props を削ることではありません。まず「Button が表現すべき責務」を決めることです。Button が担うのは、ユーザーの操作を実行することです。操作の重要度、危険度、状態、アクセシビリティは Button の責務に含まれます。一方で、画面固有の余白調整やレイアウト都合は、Button ではなく親のレイアウトが扱うべきです。

**コンポーネントAPIの判断基準は、その部品の意味に関係する差分か、画面都合の見た目調整かを分けること**です。前者は props として持つ価値があります。後者を受け入れすぎると、コンポーネントはプロダクトの判断ではなく、局所的な都合の逃げ道になります。

### 制約は開発者を縛るためではなく、迷いを減らすためにある

制約という言葉は、現場では窮屈に聞こえることがあります。特に急いでいる開発では、「少しだけ変えたいのに共通コンポーネントが許してくれない」という不満が出ます。

ただし、デザインシステムにおける制約は、開発者の創意工夫を止めるためのものではありません。**制約は、毎回判断しなくてよい範囲を増やし、レビューで見るべき論点を減らすための道具**です。

たとえばボタンの高さが決まっていれば、各画面でクリック領域や視認性を毎回検討しなくて済みます。フォーカス表示がコンポーネントに含まれていれば、キーボード操作の最低品質を画面ごとに思い出さなくて済みます。エラー表示の位置が入力欄コンポーネントで定義されていれば、フォームごとにユーザーの視線誘導を設計し直す必要が減ります。

制約が効いているコンポーネントは、使う人に「考えるな」と言っているのではありません。考えるべき場所を、より上位のプロダクト判断や例外判断に移しています。

### 拡張性は「何でもできること」ではない

拡張性を、自由に上書きできることだと捉えると、`className` や `style` を渡せることが最強の解決策に見えます。確かに、利用側が自由にCSSを当てられれば、多くの画面要件には対応できます。

しかし、その自由は長期的には保守性を下げます。コンポーネント内部を変更したとき、外側からどの画面がどんな上書きをしているか分からない。トークンを変更しても、個別CSSが残っていて反映されない。見た目の差分が、意図した拡張なのか一時的な回避策なのか判断できない。こうなると、共通コンポーネントの変更が怖くなります。

実務で扱いやすい拡張性は、無制限の上書きではなく、拡張ポイントが明確な状態です。たとえば Table なら、列定義、セルのレンダリング、空状態、ローディング状態は拡張ポイントになり得ます。一方で、行の高さ、選択状態の色、フォーカス表示の消し込みを自由にすると、一覧体験の一貫性が崩れやすくなります。

**拡張性とは、壊してよい余白を広げることではなく、壊さずに差分を出せる入口を設計すること**です。

## 実務への接続

### props の数より、組み合わせの意味を見る

コンポーネントを見直すときは、単純に props の数だけで判断しないほうがよいです。props が多くても、それぞれの意味が明確で、組み合わせが自然なら問題は小さいです。逆に props が少なくても、`custom` や `isSpecial` のような曖昧な逃げ道があると、運用はすぐに不安定になります。

確認すべきなのは、利用側がその props を選ぶときに、プロダクト上の判断ができるかどうかです。`variant="destructive"` は、取り返しにくい操作という意味を持ちます。一方で `red={true}` は、見た目の指定であり、なぜ赤いのかを説明しません。

**APIは見た目の選択肢ではなく、プロダクト上の意味を表す名前にする**と、デザインレビューとコードレビューの会話が揃いやすくなります。

### 例外を受け入れる前に、例外の種類を分ける

現場では例外要件を完全には避けられません。重要なのは、例外をすぐ props にしないことです。

まず、その例外が一度きりの画面都合なのか、複数画面で繰り返されるパターンなのかを分けます。一度きりなら、コンポーネントを広げるより、画面側の構成を変えるほうがよい場合があります。複数画面で繰り返されるなら、新しい variant、サブコンポーネント、または別コンポーネントとして定義する価値があります。

次に、その差分が意味の差分なのか、見た目の差分なのかを確認します。意味が違うならAPIに名前を与える。見た目だけなら、そもそもトークンやレイアウト側で吸収できないかを見る。この順番を持つだけで、コンポーネントは局所対応の積み重ねになりにくくなります。

### ドキュメントには「できること」だけでなく「避けること」も書く

コンポーネントのドキュメントは、props 一覧だけでは不十分です。使い方の例があっても、避けたい使い方が書かれていないと、利用側は境界を判断できません。

たとえば Button なら、「1画面に primary は原則1つ」「destructive は削除や取り消しにくい操作に使う」「画面余白の調整目的で size を変えない」といった短いルールがあるだけで、レビューの観点が揃います。

ドキュメントで明文化すべきなのは、すべてのパターンではありません。**迷いやすい境界と、破ると品質が崩れやすい制約**です。ここが書かれていると、コンポーネントの自由度は個人の好みではなく、チームの判断基準として扱えるようになります。

## 今日のまとめ

### 自由度は、利用側の便利さだけで決めない

コンポーネントの自由度を上げると、短期的には画面実装が楽になります。しかし、プロダクトとして守りたい判断まで外へ出すと、長期的にはレビュー、変更、保守が難しくなります。

**自由に変えられるコンポーネントより、どこを変えてよいかが分かるコンポーネントのほうが、チームでは使いやすい**です。

### APIは意味を表す

props は単なる設定項目ではありません。そこには、チームが何を差分として認め、何を揃えるべきだと考えているかが表れます。

色、余白、サイズをそのまま外へ出す前に、それがプロダクト上の意味を持つ差分なのかを確認する必要があります。意味があるなら名前を与え、意味がないならトークンやレイアウトの責務として扱うほうが保守しやすくなります。

### 制約は変更し続けるためにある

制約は、画面を作りにくくするためではありません。コンポーネントを安心して変更し続けるためにあります。

拡張ポイントが明確で、避けたい使い方も共有されているコンポーネントは、プロダクトが大きくなっても育てやすくなります。自由度を設計することは、将来の変更余地を守ることでもあります。
