# knowledge-input

毎日ひとつの問いを起点にした教養ノートと、SaaS開発・プロダクト運用・チーム共有に使える実務ノートを、Markdownで育てていくファイルベースの知識ベースです。

このリポジトリは、単なる個人メモではなく、あとからチームメンバーが読んでも判断材料として使えるドキュメントを残すことを目的にしています。`daily/` では歴史・哲学などの広い問いを扱い、`saas/` ではプロダクト設計、ユーザー理解、事業、運用、開発判断に直接つながるテーマを扱います。

## このリポジトリで扱うもの

- 個人の視野を広げるための教養ノート
- SaaS開発者・プロダクトチーム向けの読み物
- チーム内で前提をそろえるための共有ドキュメント
- 意思決定や改善の前に読み返せる判断材料

短いメモを大量に置くよりも、1本ずつ「問い」「背景」「具体例」「現代や実務への接続」「持ち帰り」が読める形になることを重視します。

## 使い方

教養ノートを1本作る:

```sh
make daily
```

テーマを指定して作る:

```sh
make daily TOPIC="貨幣とは何か"
```

生成されるファイルは `daily/哲学/001_テーマ.md`, `daily/歴史/001_テーマ.md` のように、`topics/topics.csv` のカテゴリ別フォルダ内で連番になります。
同じ名前のファイルがすでにある場合は上書きせず、`_v2`, `_v3` のように別名で作ります。

SaaS開発・プロダクトチーム向けのノートを作る:

```sh
make saas
```

テーマを指定してSaaS開発・プロダクトチーム向けのノートを作る:

```sh
make saas TOPIC="オンボーディングとは何か"
```

生成されるファイルは `saas/001_テーマ.md`, `saas/002_テーマ.md` のような連番形式です。

## 構成

- `daily/`: 毎日の教養ノート。`topics/topics.csv` のカテゴリに合わせてジャンル別に保存する
- `saas/`: SaaS開発者・プロダクトチーム向けのプロダクト・UX・事業・運用ノート
- `topics/topics.csv`: 自動選出用のテーマ一覧
- `topics/saas_topics.csv`: SaaS向けの自動選出用テーマ一覧
- `templates/daily.md`: 生成されるノートのテンプレート
- `templates/saas.md`: SaaS向けノートのテンプレート
- `scripts/new_daily.py`: ノート生成スクリプト
- `scripts/new_saas.py`: SaaS向けノート生成スクリプト

## ノートの基本方針

`daily/` の各ノートは次の流れで書きます。

- 今日の問い
- 読む前の見取り図
- 本文
- 現代への接続
- 今日のまとめ
- 考える問い
- 関連キーワード
- 次に読むなら

箇条書きだけで埋めるよりも、段落同士のつながりを意識して「読む前の疑問」「背景」「具体例」「現代への接続」が自然につながるように書きます。
`本文` や `現代への接続` が長くなる場合は、論点ごとに `###` の小見出しを入れます。1つのセクションを大きくしすぎず、あとから見返したときに要点を拾える構成にします。
読んだ本・記事・動画のメモを追記していくと、あとから見返せる知識ログになります。

`saas/` のノートは、`saas/README.md` の方針に沿って、冒頭で中心主張を示し、本文では考え方を説明し、最後に自分たちのプロダクトやチームで確認できる問いへ落とし込みます。SaaSノートは用語集ではなく、プロダクト設計・ユーザー理解・事業・運用・開発判断の前提をそろえるための共有ドキュメントとして扱います。

## 品質の目安

目標は、短くても長くても「読んで満足できること」です。

- 単なる用語説明ではなく、問いから始まる読み物にする
- 前の段落の疑問が、次の段落につながるように書く
- 長い本文は `###` の小見出しで分け、1つの小見出しを2〜5段落くらいに収める
- 抽象論だけでなく、生活・仕事・ニュース・人間関係の具体例に落とす
- 最後に「世界の見方が少し変わる」持ち帰りを置く
- 関連キーワードや本は、本文を読んだあとに深掘りする入口として使う

Codexに本文作成を頼むときは、次のように依頼するとよいです。

```text
daily/001_テーマ.md を、箇条書きではなく、読んで満足できる教養読み物として本文まで埋めて。
前後のつながり、具体例、現代への接続を重視して。
長いセクションは `###` の小見出しで分けて、論点ごとに読み返しやすくして。
```

SaaSやチーム共有向けの記事を頼むときは、次のように依頼するとよいです。

```text
saas/次のテーマ.md を、SaaS開発者・プロダクトチーム向けに読んで満足できる共有ドキュメントとして本文まで埋めて。
中心主張を絞り、プロダクト設計、ユーザー理解、事業、運用、開発判断への接続を明確にして。
最後は自分たちのプロダクトやチームで確認できる問いに落とし込んで。
```
