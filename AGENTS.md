# Codex Instructions

このリポジトリで作業するときは、作業開始時に必ず次の2つを参照する。

- `README.md`: リポジトリ全体の構成、ノート生成方法、liberal-arts/ と saas/ の基本方針を確認する
- `saas/README.md`: SaaSノートの狙い、文章構成、品質基準、Codexへの依頼例を確認する

特に `liberal-arts/`、`saas/`、`design-system/`、`frontend-architecture/`、`product-ui-ux-patterns/`、`accessibility/` のノートを作成・更新するときは、上記READMEの方針に加えて、次を確認してから作業する。

- 該当ジャンルの `README.md`
- 対応する `topics/*.csv`
- 使用する `templates/*.md`
- 既存の近い番号の記事、または近いテーマの記事

新しいジャンルや `topics/*.csv` のテーマを追加・更新するときは、既存の全トピックCSVを確認し、他ジャンルと主題が重ならないようにする。似た題材を扱う場合でも、ジャンルごとの役割を分ける。たとえば SaaS はプロダクト価値・事業・運用・開発判断、Product UI / UX Patterns は画面構造・操作パターン・情報設計、Design System はコンポーネント・トークン・運用、Frontend Architecture は実装設計・責務分離、Accessibility は利用可能性・支援技術・アクセシブルな操作に寄せる。

## 作業前に確認すること

- 依頼が「新規作成」か「既存記事の本文補強」か「トピックやテンプレートの更新」かを見分ける
- 新規作成では、既存の番号・カテゴリ・ファイル名の規則を確認し、連番やカテゴリ配置を崩さない
- 既存記事の更新では、見出し構成、語調、太字の使い方、最後の問いの粒度を周辺記事と合わせる
- トピック追加では、全 `topics/*.csv` を確認し、同じ主題を別ジャンルに重複登録しない
- 実装や生成スクリプトを変更した場合は、可能な範囲で該当する `make` コマンドやスクリプトを実行して確認する

## ノート作成時の基本方針

- 既存の番号・カテゴリ・ファイル名ルールを崩さない
- テンプレートを起点にしつつ、箇条書きだけで終わらせず、読み物として成立する本文まで書く
- SaaSノートでは、中心主張を絞り、プロダクト設計・ユーザー理解・事業・運用・開発判断への接続を明確にする
- 教養ノートでは、問いから始まり、背景・具体例・現代への接続・持ち帰りが自然につながるようにする

## ジャンルごとの書き分け

- `liberal-arts/`: 歴史・哲学・テクノロジーなどの問いを、背景、具体例、現代への接続まで含めて扱う。実務ノウハウではなく、ものの見方が変わる読み物にする
- `saas/`: SaaSのプロダクト価値、ユーザー理解、事業、運用、開発判断に接続する。UIパターンや実装詳細だけに寄せすぎない
- `product-ui-ux-patterns/`: 画面構造、操作パターン、情報設計に寄せる。事業一般論よりも、ユーザーが画面上で何を見て、どう判断し、どこで詰まるかを扱う
- `design-system/`: コンポーネント、デザイントークン、設計原則、変更管理、チーム運用に寄せる。個別画面の改善案だけで終わらせない
- `frontend-architecture/`: 実装設計、責務分離、状態管理、API境界、テスト、技術的負債に寄せる。UIの見た目より、変更し続けられる構造を扱う
- `accessibility/`: 利用可能性、支援技術、キーボード操作、フォーム、エラー、チーム運用に寄せる。善意や配慮の話だけでなく、プロダクト品質として扱う

## 避けること

- テンプレートの見出しだけを埋め、本文が箇条書きのメモで止まること
- 同じ中心主張を、本文、接続、まとめで言い換えて繰り返すこと
- 具体例を浅く並べすぎて、代表例から判断基準が見えなくなること
- SaaS、Product UI / UX Patterns、Design System、Frontend Architecture、Accessibility の境界を曖昧にし、同じ切り口の記事を複数ジャンルに作ること
- 読者が次に確認・改善できない抽象的な問いだけで終わること

## 文章の強調と読みやすさ

- Markdownの `**太字**` は積極的に使ってよい。多少教材っぽくなってもよいので、読者が後から見返したときに重要点を拾いやすい文章にする
- 太字にする対象は、中心主張、定義、重要な対比、判断基準、実務や現代への接続点を優先する
- ただし、太字だらけで要点が埋もれないように、1つの段落では特に重要な1箇所を中心に強調する
- 小見出しは単語だけでなく、問いや主張が伝わる形にする。見出しだけを追っても論点の流れが分かるようにする
- 長めの章では、必要に応じて章末に「ここで押さえること」に相当する短い要点文を置き、その章で理解してほしいことを明確にする
- 誤解しやすい点、よくある失敗、初心者が混同しやすい概念があれば、本文中で明示する
- 具体例は複数を浅く並べすぎず、理解を助ける代表例を1つ深めに扱う
- 重要語は、初出時に `**重要語**` のように太字で示し、必要に応じて短い定義を添える
- 最後の問いは、読者が実際に確認・改善・内省へ移せる具体的な問いにする

## 仕上げの確認

- 見出しだけを追っても、問い、背景、中心主張、実務や現代への接続、持ち帰りの流れが分かるか
- 太字が中心主張、定義、重要な対比、判断基準に使われているか
- 代表例が本文の理解を深めており、例の数で水増ししていないか
- 最後の問いが、読者の確認、改善、内省に移せる具体性を持っているか
- 関連キーワードや「次に読むなら」「次に深掘りするなら」が、本文の続きとして自然か
