# Frontend Architecture

SaaSプロダクトのフロントエンドを変更し続けるための設計判断を整理するノートを置く場所です。

扱うテーマは、状態管理、責務分離、フォーム、API境界、テスト、パフォーマンス、技術的負債です。

トピック一覧は `topics/frontend_architecture_topics.csv` を参照します。

## 使い方

```sh
make frontend-architecture
```

テーマを指定する場合:

```sh
make frontend-architecture TOPIC="状態管理はなぜ複雑になるのか"
```

## 書き方の方針

- `templates/practical.md` を起点にし、実装設計、責務分離、状態管理、API境界、テスト、技術的負債のどこに効く話かを明確にする
- UIの見た目や事業一般論に寄せすぎず、**変更し続けられるフロントエンド構造をどう作るか**を中心に扱う
- 具体例は、フォーム、一覧、詳細、権限、非同期処理、キャッシュ、テスト、共通化など、実装判断が割れやすい題材を選ぶ
- 「実務への接続」では、コードの責務、境界、変更時の影響範囲、テストで守るべき振る舞いに落とし込む
- 最後の問いは、既存コード、変更履歴、レビュー、障害、テスト不足を確認できる粒度にする
